# v0.3 真实场景输出质量测试用例

日期：2026-06-20  
目标版本：v0.3  
测试对象：`product-spec-mcp`，重点验证 `product_spec_assist` 统一入口和 6 个细分工具的输出质量。  
测试人群：小白 / 独立开发者。  
测试范围：只测 MCP 输出质量，不要求实际交给 coding agent 完整实现项目。

## 1. 测试目标

v0.3 上线前需要确认：

- 小白用户用一句自然语言输入时，`product_spec_assist` 能正确路由。
- MCP 不会过度假设，不会把模糊需求包装成“可直接开发”的假规格。
- 追问问题能覆盖目标用户、平台、核心流程、数据、权限、验收等关键缺口。
- 架构建议能识别纯前端、后端、鉴权、支付、AI Key 等风险。
- UI 口语能稳定翻译成前端修改语言。
- Debug 指引能要求用户提供真实证据，而不是凭空猜原因。
- 验收清单能作为后续开发质量门槛。
- `structuredContent` 字段稳定、可被 agent 消费。

## 2. 通用测试方法

每个用例都按以下方式检查：

1. 先调用 `product_spec_assist`。
2. 观察 `routedIntent.scenario`、`selectedTool`、`executed`、`nextAction`。
3. 根据统一入口建议，再调用对应细分工具。
4. 同时检查 Markdown 和 `structuredContent`。
5. 记录失败点：路由错误、追问缺口、默认假设误导、架构风险漏报、验收清单不可用。

通用通过标准：

- 不抛异常。
- Markdown 可读。
- `structuredContent` 存在且字段稳定。
- 输出没有明显自相矛盾。
- 不把低信息输入误判成可直接开发。
- 不对小白用户输出过度技术化但无行动指引的内容。

通用失败标准：

- 用户一句话被路由到错误场景。
- 缺少关键追问却直接生成正式规格。
- 静态官网被推荐数据库/API/后台。
- 涉及支付或 AI Key 却没有风险提示。
- Debug 输出没有要求 Console、Network、日志、报错截图等证据。
- 验收标准只有泛泛而谈，不能检查真实交付。

## 3. 测试用例一：小白做报名/预约系统

### 3.1 场景目的

验证“从模糊产品想法到可执行规格”的主链路。

覆盖能力：

- `product_spec_assist` 路由 `build_product`
- `spec_interrogate`
- `spec_compile`
- `architecture_decide`
- `acceptance_generate`
- readiness 评分
- 追问质量
- 后端/数据库/权限判断
- 表单类验收标准

### 3.2 用户画像

用户是独立开发者，第一次做一个给线下活动用的报名系统，不清楚需要后端、数据库、权限、导出、审核这些概念。

### 3.3 初始输入

```text
我想做一个活动报名系统，用户可以提交姓名电话和报名信息，我能在后台看到，最好还能导出 Excel。
```

### 3.4 第一步：调用统一入口

工具：

```text
product_spec_assist
```

输入：

```json
{
  "message": "我想做一个活动报名系统，用户可以提交姓名电话和报名信息，我能在后台看到，最好还能导出 Excel。",
  "preferred_platform": "web",
  "strictness": "normal"
}
```

期望 `structuredContent`：

```ts
{
  routedIntent: {
    scenario: "build_product",
    confidence: number
  },
  selectedTool: "spec_interrogate",
  executed: true,
  nextAction: {
    type: "answer_questions" | "compile_spec",
    suggestedTool: "spec_interrogate" | "spec_compile"
  }
}
```

关键断言：

- scenario 必须是 `build_product`。
- selectedTool 优先为 `spec_interrogate`。
- 输出应识别这是表单/报名/后台查看/导出场景。
- 不应直接说“可以纯前端实现”。
- 不应直接生成正式开发 Prompt，除非明确标注仍有默认假设。

### 3.5 第二步：调用需求追问

工具：

```text
spec_interrogate
```

输入：

```json
{
  "raw_idea": "我想做一个活动报名系统，用户可以提交姓名电话和报名信息，我能在后台看到，最好还能导出 Excel。",
  "scenario": "build_product",
  "target_platform": "web",
  "strictness": "normal"
}
```

期望追问覆盖：

- 活动报名是否需要登录。
- 后台谁能访问。
- 是否需要审核报名。
- 报名字段有哪些。
- 是否需要防重复提交。
- 是否需要导出 Excel。
- 数据保存在哪里。
- 是否需要短信/邮件通知。
- 手机端是否是主要入口。
- 验收标准是什么。

关键断言：

- readiness 不应过高；如果信息仍缺失，应是 `Draft Ready` 或 `Not Ready`。
- `missingFields` 至少包含权限、工作流、验收或数据相关缺口。
- `defaultAssumptions` 必须明确，不应隐藏假设。

### 3.6 第三步：补充回答后编译规格

模拟用户补充：

```json
{
  "target_user": "参加线下活动的普通用户",
  "platform": "web",
  "core_features": ["报名表单", "后台查看报名", "导出 Excel", "防重复提交"],
  "data_persistence": true,
  "user_roles": ["报名用户", "管理员"],
  "backend_need": true,
  "workflow": "用户提交报名 -> 系统保存 -> 管理员后台查看 -> 管理员导出 Excel",
  "success_criteria": [
    "用户能在手机端完成报名",
    "管理员能看到报名列表",
    "管理员能导出 Excel",
    "重复提交能被阻止"
  ],
  "non_goals": ["暂不做在线支付", "暂不做短信通知"]
}
```

工具：

```text
spec_compile
```

输入：

```json
{
  "raw_idea": "活动报名系统，用户提交姓名电话和报名信息，管理员后台查看并导出 Excel。",
  "answers": {
    "target_user": "参加线下活动的普通用户",
    "platform": "web",
    "core_features": ["报名表单", "后台查看报名", "导出 Excel", "防重复提交"],
    "data_persistence": true,
    "user_roles": ["报名用户", "管理员"],
    "backend_need": true,
    "workflow": "用户提交报名 -> 系统保存 -> 管理员后台查看 -> 管理员导出 Excel",
    "success_criteria": [
      "用户能在手机端完成报名",
      "管理员能看到报名列表",
      "管理员能导出 Excel",
      "重复提交能被阻止"
    ],
    "non_goals": ["暂不做在线支付", "暂不做短信通知"]
  },
  "allow_assumptions": true,
  "min_readiness_score": 70
}
```

期望输出：

- `mode` 应为 `formal` 或高质量 `draft`。
- spec 中必须包含用户、平台、核心功能、数据、后台、导出、风险边界、暂不包含。
- API 不应出现 `POST /api/核心功能` 这种空泛内容。
- 如果建议数据库，应符合小型系统场景，例如 SQLite/PostgreSQL 二选一，并说明 MVP 和正式版差异。

### 3.7 第四步：架构决策

工具：

```text
architecture_decide
```

输入：

```json
{
  "product_type": "活动报名系统",
  "platform": "web",
  "features": ["报名表单", "后台查看报名", "导出 Excel", "防重复提交", "管理员权限"],
  "commercial_intent": false,
  "expected_users": "small_team"
}
```

期望输出：

- `needBackend = true`
- `needAdmin = true`
- `needLogging = true`
- `canBeFrontendOnly = false`
- 应推荐轻量后端和数据库。
- 不应推荐支付相关内容。

### 3.8 第五步：验收标准

工具：

```text
acceptance_generate
```

输入：

```json
{
  "product_type": "活动报名系统",
  "features": ["表单", "提交", "后台管理", "导出"],
  "platform": "web",
  "has_backend": true,
  "has_payment": false,
  "has_auth": true
}
```

期望验收项：

- 表单必填校验。
- 提交中 loading。
- 提交成功/失败提示。
- 防重复提交。
- 后台接口鉴权。
- 管理员权限不能只靠前端。
- 移动端不能横向溢出。
- 导出功能可用。
- 错误日志足够排查。

## 4. 测试用例二：小白做个人展示官网，并反复说“高级一点”

### 4.1 场景目的

验证 MCP 不会过度工程化静态展示站，同时能把小白 UI 口语翻译成可执行前端修改语言。

覆盖能力：

- `product_spec_assist` 路由 `build_product`
- 静态展示类架构判断
- `ui_translate`
- `acceptance_generate`
- launch 信息缺口检查
- 避免错误推荐后端/数据库/API

### 4.2 用户画像

用户是自由职业设计师，想做一个个人作品展示官网，不懂前后端区别，只会用“高级一点”“上面那块”“手机上乱了”描述 UI。

### 4.3 初始输入

```text
我想做一个个人作品展示网站，放我的介绍、作品图片、联系方式，看起来高级一点，手机上也要好看。
```

### 4.4 第一步：统一入口路由

工具：

```text
product_spec_assist
```

输入：

```json
{
  "message": "我想做一个个人作品展示网站，放我的介绍、作品图片、联系方式，看起来高级一点，手机上也要好看。",
  "preferred_platform": "web",
  "strictness": "light"
}
```

期望：

- scenario = `build_product`
- selectedTool = `spec_interrogate`
- 输出识别为展示站/作品集/landing page。
- 应提示这是可纯前端的候选场景。
- 不应默认推荐登录、数据库、后台。

### 4.5 第二步：架构决策

工具：

```text
architecture_decide
```

输入：

```json
{
  "product_type": "个人作品展示网站",
  "platform": "web",
  "features": ["个人介绍", "作品展示", "联系方式", "响应式布局"],
  "commercial_intent": false,
  "expected_users": "individual"
}
```

期望输出：

- `canBeFrontendOnly = true`
- `needBackend = false`
- `needSeparation = false`
- `needAuth = false`
- `needAdmin = false`
- 推荐静态托管/CDN。
- 不应推荐数据库。
- 不应生成 API 设计。

失败判定：

- 出现 PostgreSQL、SQLite、API、后台管理、JWT 等非必要建议，除非明确解释为非当前需求。

### 4.6 第三步：UI 口语翻译

工具：

```text
ui_translate
```

输入一：

```json
{
  "description": "首页上面那块太普通了，高级一点",
  "current_page": "首页"
}
```

期望：

- 识别 Header/Hero/视觉层级/字体/留白/配色等方向。
- 生成可给前端或 coding agent 的修改 Prompt。
- 不要只说“美化页面”。

输入二：

```json
{
  "description": "手机上图片和文字有点乱，按钮也挤在一起",
  "current_page": "作品页"
}
```

期望：

- 识别响应式布局问题。
- 包含 mobile-first、media query、spacing、grid/flex、touch target 等术语或等价中文解释。
- 给出具体修改区域和方向。

### 4.7 第四步：上线前检查

工具：

```text
product_spec_assist
```

输入：

```json
{
  "message": "这个个人作品网站我想上线了，需要注意什么？",
  "preferred_platform": "web"
}
```

期望：

- scenario = `launch`
- 不应声称“可以直接上线”。
- 应要求或提醒：
  - 域名。
  - 静态托管平台。
  - 图片压缩。
  - 移动端检查。
  - 表单/联系方式是否可用。
  - SEO title/description。
  - 404 页面。
  - 访问速度。

### 4.8 第五步：验收标准

工具：

```text
acceptance_generate
```

输入：

```json
{
  "product_type": "个人作品展示网站",
  "features": ["展示", "图片", "联系方式", "响应式"],
  "platform": "web",
  "has_backend": false,
  "has_payment": false,
  "has_auth": false
}
```

期望验收项：

- 桌面端和移动端正常显示。
- 移动端不能横向溢出。
- 图片正常加载。
- 页面加载时间可接受。
- 联系方式可点击或可复制。
- 所有链接可点击。
- 不应出现后端接口、权限、支付验收项。

## 5. 测试用例三：小白做 AI 工具，涉及 API Key、付费和报错排查

### 5.1 场景目的

验证 MCP 对高风险独立开发项目的判断能力：AI Key 不能前端暴露、支付不能只靠前端、Debug 要收集证据。

覆盖能力：

- `product_spec_assist` 路由 `build_product`
- `architecture_decide` 高风险判断
- `debug_guide`
- `acceptance_generate`
- AI Key 风险
- 支付风险
- 后端代理、用量统计、限流、日志

### 5.2 用户画像

用户是独立开发者，想做一个 AI 文案生成小工具，准备商业化，但不懂 API Key、支付回调、限流和日志。

### 5.3 初始输入

```text
我想做一个 AI 文案生成工具，用户输入产品介绍，系统帮他生成小红书文案。以后我想收费，可以按次数卖套餐。
```

### 5.4 第一步：统一入口路由

工具：

```text
product_spec_assist
```

输入：

```json
{
  "message": "我想做一个 AI 文案生成工具，用户输入产品介绍，系统帮他生成小红书文案。以后我想收费，可以按次数卖套餐。",
  "preferred_platform": "web",
  "strictness": "normal"
}
```

期望：

- scenario = `build_product`
- selectedTool = `spec_interrogate`
- 输出识别 AI、商业化、次数套餐。
- 必须提示需要后端或后端代理。
- 不应建议把 AI API Key 放前端。
- 不应忽略支付/套餐/用量统计。

### 5.5 第二步：架构决策

工具：

```text
architecture_decide
```

输入：

```json
{
  "product_type": "AI 文案生成 SaaS",
  "platform": "web",
  "features": ["AI", "GPT", "用户登录", "套餐购买", "按次数扣减", "支付", "历史记录"],
  "commercial_intent": true,
  "expected_users": "small_team"
}
```

期望输出：

- `needBackend = true`
- `needAuth = true`
- `needLogging = true`
- `paymentRisk = true`
- `aiKeyRisk = true`
- 推荐后端代理 AI API。
- 推荐订单/套餐/用量记录。
- 提醒支付状态必须以后端回调或订单查询确认。
- 提醒支付金额必须后端计算。
- 提醒需要限流、用量统计、错误日志。

失败判定：

- 没有 AI Key 风险。
- 没有支付回调风险。
- 允许纯前端。
- 没有鉴权和用量记录建议。

### 5.6 第三步：需求追问

工具：

```text
spec_interrogate
```

输入：

```json
{
  "raw_idea": "AI 文案生成工具，用户输入产品介绍，系统生成小红书文案，未来按次数卖套餐。",
  "scenario": "build_product",
  "target_platform": "web",
  "strictness": "grill"
}
```

期望追问覆盖：

- 使用哪家 AI 服务。
- API Key 如何保存。
- 是否需要登录。
- 套餐如何定义。
- 次数如何扣减。
- 支付方式。
- 生成失败是否扣次数。
- 是否保存历史记录。
- 是否有内容安全限制。
- 是否需要限流。
- 成功标准和验收方式。

### 5.7 第四步：Debug 指引

模拟用户报错：

```text
我点生成之后一直转圈，最后提示失败，但是我不知道哪里错了。
```

工具：

```text
debug_guide
```

输入：

```json
{
  "platform": "web",
  "error_description": "AI 文案生成工具点击生成后一直 loading，最后提示失败。",
  "current_info": {
    "browser": "Chrome",
    "user_action": "点击生成按钮"
  }
}
```

期望输出：

- 要求提供 Console 红色报错。
- 要求提供 Network 中失败请求、status code、response。
- 要求提供后端日志或 request_id。
- 提醒检查 AI API Key、后端代理、超时、额度、限流。
- 提醒不要只看前端 toast。
- `knownInfo` 应标记已提供项。
- `missingRequiredInfo` 应列出缺失证据。

失败判定：

- 直接猜测“可能网络问题”，但不要求证据。
- 不要求 Network/Console/后端日志。
- 不涉及 AI API 相关排查。

### 5.8 第五步：验收标准

工具：

```text
acceptance_generate
```

输入：

```json
{
  "product_type": "AI 文案生成 SaaS",
  "features": ["AI", "登录", "支付", "套餐", "历史记录"],
  "platform": "web",
  "has_backend": true,
  "has_payment": true,
  "has_auth": true
}
```

期望验收项：

- AI API Key 不暴露在前端。
- 所有敏感接口有鉴权。
- 支付金额后端计算。
- 支付状态以后端回调或订单查询确认。
- 订单状态幂等。
- 生成失败有明确提示。
- loading 状态可见。
- 生成历史可查看。
- 错误日志可追踪。
- 权限校验不能只靠前端。
