# v0.3 实施方案：统一入口与 intentRouter 接入

日期：2026-06-20  
目标版本：v0.3  
主题：接入 `intentRouter`，新增统一入口工具，降低用户选择 MCP tool 的成本。

前置条件：v0.2 已完成，现有工具具备稳定的 `structuredContent`，方便统一入口复用结构化结果。

## 1. 背景

v0.1 暴露了 6 个工具：

- `spec_interrogate`
- `spec_compile`
- `architecture_decide`
- `ui_translate`
- `debug_guide`
- `acceptance_generate`

这对熟悉项目的人清晰，但对新用户和 AI 客户端不够友好。用户通常只会输入一句话：

- “我想做一个报名系统”
- “首页高级一点”
- “小程序白屏了”
- “这个项目能不能上线”

他们不应该先理解 6 个工具的区别。

v0.3 的目标是：

> 新增一个统一入口，根据用户原话自动判断场景，并给出下一步或直接调用对应能力。

## 2. 目标

新增 MCP tool：

```text
product_spec_assist
```

能力：

- 接收用户自然语言输入。
- 使用 `intentRouter` 判断场景。
- 根据场景调用一个最合适的现有 engine。
- 返回 Markdown + `structuredContent`。
- 告诉用户下一步建议调用什么。

## 3. 非目标

v0.3 不做：

- 不做多轮状态机。
- 不做跨会话记忆。
- 不自动连续调用多个 MCP tool。
- 不写入文件。
- 不做规则 profile。
- 不接 Jira、Linear、GitHub Projects。
- 不做完整 spec-driven workflow。

重要边界：

> v0.3 是“轻量路由入口”，不是 Kiro/spec-coding-mcp 那种完整 SDD 工作流。

## 4. 工具输入设计

建议新增 schema：

```text
src/schemas/productSpecAssist.schema.ts
```

输入：

```ts
{
  message: string;
  known_context?: Record<string, unknown>;
  preferred_platform?: "web" | "mini_program" | "app" | "backend" | "unknown";
  strictness?: "light" | "normal" | "grill";
  auto_execute?: boolean;
}
```

字段说明：

- `message`：用户原话，必填。
- `known_context`：已有上下文，可透传给需求追问/规格编译。
- `preferred_platform`：用户已知平台。
- `strictness`：追问强度。
- `auto_execute`：是否允许自动调用对应 engine。默认 `true`。

第一版建议 `auto_execute` 默认 true，但只允许执行一个 engine，不做链式执行。

## 5. 路由输出设计

建议结构：

```ts
{
  routedIntent: {
    intent: string,
    scenario: "build_product" | "modify_ui" | "debug" | "launch" | "unknown",
    confidence: number
  },
  selectedTool: string | null,
  executed: boolean,
  result?: unknown,
  nextAction: {
    type:
      | "answer_questions"
      | "compile_spec"
      | "translate_ui"
      | "provide_debug_info"
      | "review_launch_readiness"
      | "choose_tool_manually";
    message: string;
    suggestedTool?: string;
  }
}
```

Markdown 输出应包含：

- 识别到的场景。
- 置信度。
- 已执行的动作。
- 关键结果摘要。
- 下一步建议。

## 6. 路由规则

基于现有 `src/core/intentRouter.ts`：

| scenario | 默认动作 | selectedTool |
| --- | --- | --- |
| `build_product` | 运行需求追问 | `spec_interrogate` |
| `modify_ui` | 运行 UI 术语翻译 | `ui_translate` |
| `debug` | 运行 Debug 指引 | `debug_guide` |
| `launch` | 第一版先输出上线信息缺口，不新增复杂工具 | `spec_interrogate` 或 `acceptance_generate` |
| `unknown` | 返回可选工具和澄清问题 | `null` |

### 6.1 build_product

调用：

```ts
calculateReadiness(message, known_context)
generateClarification(...)
formatInterrogateResult(...)
```

不要直接 `spec_compile`，除非 readiness 已经 `Build Ready` 且上下文足够完整。第一版建议保守：统一先追问。

### 6.2 modify_ui

调用：

```ts
translateUiDescription(message)
formatUiTranslateResult(...)
```

输出下一步建议：把生成的修改 Prompt 交给代码 Agent 执行。

### 6.3 debug

平台判断：

- 如果 message 包含“小程序”，platform = `mini_program`
- 包含“后端/接口/500/API”，platform = `backend`
- 包含“构建/build/编译”，platform = `build`
- 否则默认 `web`

调用：

```ts
generateDebugGuide(platform, message, known_context)
formatDebugGuideResult(...)
```

### 6.4 launch

当前没有专门 launch engine。第一版不要硬造完整上线工具。

建议：

- 输出“上线前缺口检查”。
- 推荐用户补充：平台、部署方式、域名、环境变量、数据迁移、验收结果、日志和回滚方案。
- 可复用 `acceptance_generate` 生成基础验收清单，但不要声称已完成上线评估。

### 6.5 unknown

返回工具选择建议：

- 想描述新产品：`spec_interrogate`
- 想生成开发规格：`spec_compile`
- 想改 UI：`ui_translate`
- 程序报错：`debug_guide`
- 想生成验收标准：`acceptance_generate`

## 7. 文件级实施建议

新增：

```text
src/schemas/productSpecAssist.schema.ts
src/tools/productSpecAssist.ts
src/core/assistEngine.ts
tests/productSpecAssist.test.ts
```

修改：

```text
src/server.ts
README.md
```

### 7.1 `assistEngine.ts`

建议职责：

- 调用 `routeIntent`。
- 根据 scenario 调用对应 core engine。
- 生成统一结构化结果。
- 生成统一 Markdown。

不要把路由逻辑写在 `tools/productSpecAssist.ts` 里。

### 7.2 `productSpecAssist.ts`

只做：

- 注册工具。
- parse input。
- 调用 `assistEngine`。
- 返回 `content` + `structuredContent`。

### 7.3 `server.ts`

注册新工具：

```ts
registerProductSpecAssist(server);
```

工具数量从 6 个变成 7 个。

## 8. 测试方案

新增测试覆盖：

### 8.1 路由测试

输入：

```text
我想做一个报名系统，学生可以提交资料，后台审核
```

期望：

- scenario = `build_product`
- selectedTool = `spec_interrogate`
- executed = true
- result 包含 readiness

输入：

```text
首页上面那块高级一点
```

期望：

- scenario = `modify_ui`
- selectedTool = `ui_translate`
- result 包含 frontendTerms

输入：

```text
小程序打开以后白屏
```

期望：

- scenario = `debug`
- selectedTool = `debug_guide`
- platform = `mini_program`

输入：

```text
我想上线这个项目
```

期望：

- scenario = `launch`
- 不声称可直接上线
- nextAction 要求补充上线信息

输入：

```text
帮我看看
```

期望：

- scenario = `unknown`
- executed = false 或只返回澄清
- nextAction.type = `choose_tool_manually`

### 8.2 回归测试

确保新增统一入口不影响 6 个原工具现有行为。

## 11. v0.3 完成定义

v0.3 完成需满足：

- 新增 `product_spec_assist` MCP tool。
- `intentRouter` 被实际接入。
- 至少覆盖 build_product、modify_ui、debug、launch、unknown 五类输入。
- 每类输入有测试。
- 统一入口返回 Markdown + `structuredContent`。
- 6 个原有工具保持兼容。
- README 明确推荐新用户先用统一入口。

## 12. 风险与注意事项

- 不要把统一入口做成复杂 agent loop。
- 不要在路由不确定时强行生成规格。
- 不要为 launch 场景临时写一堆不稳定上线逻辑；第一版只做上线信息缺口检查。
- 不要移除原 6 个工具。统一入口是补充，不是替代。
- 不要把规则 profile 混进 v0.3。

## 13. 与 v0.4 的衔接

规则系统产品化接口建议延后到 v0.4。

v0.3 完成后，再基于统一入口观察哪些规则最需要定制：

- 默认技术栈。
- 数据库偏好。
- 验收标准。
- Debug playbook。
- 风险红线。

届时再设计：

```text
.product-spec-mcp/rules.json
```

不要在 v0.3 提前实现，以免影响统一入口的稳定交付。
