# v0.2 实施方案：结构化输出与 Agent 可读结果

日期：2026-06-20  
目标版本：v0.2  
主题：为现有 MCP tools 增加 `structuredContent` 与 `outputSchema`。

## 1. 背景

v0.1 的工具输出主要是 Markdown：

```ts
return {
  content: [{ type: "text", text: markdown }],
};
```

这对人类阅读友好，但对 MCP 客户端、AI agent、未来 UI、自动化测试和商业化统计都不友好。

v0.2 的核心目标是：

> 保留 Markdown 可读性，同时返回稳定的机器可读 JSON。

前置条件：v0.1 的 `npm run typecheck`、`npm test`、`npm run build` 已稳定通过。

## 2. 目标

v0.2 需要完成：

- 每个核心 tool 返回 `structuredContent`。
- 优先为 3 个关键 tool 增加 `outputSchema`：
  - `spec_interrogate`
  - `spec_compile`
  - `acceptance_generate`
- 保持现有 Markdown 输出兼容。
- 增加结构化输出测试。
- 更新 README，说明人类读 Markdown、agent 读 `structuredContent`。

## 3. 非目标

v0.2 不做：

- 不新增统一入口工具。
- 不接入 `intentRouter`。
- 不做规则 profile。
- 不改 MCP transport。
- 不做云端服务。
- 不重写规则引擎。

## 4. 设计原则

### 4.1 Markdown 和 JSON 双轨

所有工具结果保持：

```ts
{
  content: [{ type: "text", text: markdown }],
  structuredContent: { ... }
}
```

不要删除 Markdown。很多 MCP 客户端和用户仍然依赖文本显示。

### 4.2 结构字段必须稳定

字段名一旦进入 v0.2，就当作公开接口处理。

要求：

- 使用英文 snake_case 或 camelCase 二选一。
- 建议沿用当前内部对象的 camelCase。
- 不把整段 Markdown 塞进结构字段里。
- 不返回函数、Map、Set、Date object 等不稳定结构。
- 所有数组字段即使为空也返回空数组。

### 4.3 先覆盖关键工具

优先级：

1. `spec_interrogate`
2. `spec_compile`
3. `acceptance_generate`
4. `architecture_decide`
5. `ui_translate`
6. `debug_guide`

如果时间有限，先完成前三个；但最终 v0.2 完成定义要求 6 个都具备 `structuredContent`。

## 5. 建议输出结构

### 5.1 `spec_interrogate`

建议结构：

```ts
{
  readiness: {
    score: number,
    status: "Not Ready" | "Draft Ready" | "Build Ready",
    fields: Record<string, {
      weight: number,
      present: boolean,
      value?: string
    }>
  },
  clarification: {
    missingFields: string[],
    questions: Array<{
      field: string,
      question: string,
      whyImportant: string,
      options: string[],
      defaultAssumption: string,
      priority: string
    }>,
    defaultAssumptions: Record<string, string>
  },
  recommendation: {
    canProceed: boolean,
    suggestedNextTool: "spec_compile" | "spec_interrogate",
    reason: string
  }
}
```

推荐规则：

- `Build Ready`：`canProceed=true`，`suggestedNextTool=spec_compile`
- `Draft Ready`：`canProceed=true`，但 reason 标明需要默认假设
- `Not Ready`：`canProceed=false`，`suggestedNextTool=spec_interrogate`

### 5.2 `spec_compile`

建议结构：

```ts
{
  mode: "not_ready" | "draft" | "formal",
  readiness: {
    score: number,
    status: string
  },
  spec?: {
    productGoal: string,
    targetUser: string,
    platform: string,
    coreFeatures: string[],
    dataModel: string,
    architecture: string,
    apiDesign: string,
    riskBoundaries: string[],
    nonGoals: string[],
    successCriteria: string[]
  },
  confirmation?: {
    items: string[]
  },
  clarification?: {
    missingFields: string[],
    questions: unknown[],
    defaultAssumptions: Record<string, string>
  },
  nextAction: {
    type: "answer_questions" | "confirm_spec" | "start_build",
    message: string
  }
}
```

注意：

- `not_ready` 时不返回伪正式 spec。
- `draft` 时必须保留默认假设和风险提示。
- `formal` 时可推荐进入开发。

### 5.3 `acceptance_generate`

建议结构：

```ts
{
  productType: string,
  platform: string,
  categories: Array<{
    category: string,
    items: string[]
  }>,
  definitionOfDone: string[],
  checklist: Array<{
    id: string,
    category: string,
    text: string,
    required: boolean
  }>
}
```

说明：

- `categories` 保持现有结构。
- 新增扁平化 `checklist`，方便 agent 逐项检查。
- `id` 可用稳定 slug，例如 `page-acceptance-1`。

### 5.4 `architecture_decide`

建议结构：

```ts
{
  decision: {
    canBeFrontendOnly: boolean,
    needBackend: boolean,
    needSeparation: boolean,
    recommendedDatabase: string,
    needAuth: boolean,
    needAdmin: boolean,
    needLogging: boolean,
    paymentRisk: boolean,
    aiKeyRisk: boolean,
    mvpSuggestion: string,
    productionSuggestion: string,
    reasoning: string[]
  },
  riskLevel: "low" | "medium" | "high",
  blockers: string[]
}
```

### 5.5 `ui_translate`

建议结构：

```ts
{
  translation: {
    originalDescription: string,
    identifiedIntent: string,
    frontendTerms: string[],
    modificationPrompt: string,
    suggestedComponent: string,
    codeHints: string[]
  },
  confidence: "low" | "medium" | "high"
}
```

### 5.6 `debug_guide`

建议结构：

```ts
{
  guide: {
    platform: string,
    requiredInfo: Array<{
      field: string,
      description: string,
      howToGet: string,
      priority: string
    }>,
    knownInfo: Record<string, unknown>,
    checklist: string[],
    commonIssues: string[],
    troubleshootingSteps: string[]
  },
  missingRequiredInfo: string[],
  canDiagnoseNow: boolean
}
```

## 6. 文件级实施建议

### 6.1 新增 output schema 文件

建议新增：

```text
src/schemas/outputs/
├── acceptanceGenerate.output.ts
├── architectureDecide.output.ts
├── debugGuide.output.ts
├── specCompile.output.ts
├── specInterrogate.output.ts
└── uiTranslate.output.ts
```

每个文件导出：

```ts
export const XxxOutputSchema = z.object({ ... });
export type XxxOutput = z.infer<typeof XxxOutputSchema>;
```

### 6.2 新增结构化结果 builder

建议新增：

```text
src/core/structuredResultBuilder.ts
```

或按领域拆分：

```text
src/core/structured/
├── acceptance.ts
├── architecture.ts
├── debug.ts
├── spec.ts
└── ui.ts
```

原则：

- 不把结构化转换逻辑塞进 `tools/*.ts`。
- `tools/*.ts` 仍然只做注册、调用 engine、返回结果。

### 6.3 修改 tools 注册

为声明结构化输出契约的工具在 `registerTool` metadata 中增加 `outputSchema`。v0.2 至少覆盖 `spec_interrogate`、`spec_compile`、`acceptance_generate`；其他工具可以先只返回 `structuredContent`，再逐步补齐 `outputSchema`。

伪代码：

```ts
server.registerTool(
  "spec_interrogate",
  {
    title: "...",
    description: "...",
    inputSchema: SpecInterrogateInputSchema.shape,
    outputSchema: SpecInterrogateOutputSchema.shape,
  },
  async (input) => {
    ...
    return {
      content: [{ type: "text", text: markdown }],
      structuredContent,
    };
  }
);
```

如果 SDK 类型不接受 Zod shape 或再次触发类型膨胀，应显式收窄类型，不要牺牲 `typecheck`。

注意：MCP SDK 支持 `outputSchema` 和 `structuredContent`；一旦某个工具声明了 `outputSchema`，该工具返回的 `structuredContent` 必须与 schema 对齐。

## 7. 测试方案

新增或扩展测试：

```text
tests/structuredOutput.test.ts
```

重点断言：

- 每个工具 handler 返回 `structuredContent`。
- `spec_interrogate.structuredContent.readiness.score` 是 number。
- `spec_interrogate.structuredContent.clarification.questions` 是数组。
- `spec_compile` 在 `not_ready` 时不返回正式 `spec`。
- `spec_compile` 在 `formal` 时返回 `spec.coreFeatures`。
- `acceptance_generate` 返回扁平化 checklist。
- 输出结构可被对应 Zod output schema parse。

如现有测试只测 core 函数，建议增加工具级测试，至少覆盖返回对象结构。

## 10. v0.2 完成定义

v0.2 完成需满足：

- 6 个工具全部返回 `structuredContent`。
- 至少 3 个关键工具声明 `outputSchema`：`spec_interrogate`、`spec_compile`、`acceptance_generate`。
- 测试覆盖结构字段。
- `npm run typecheck && npm test && npm run build` 全部通过。
- README 解释结构化输出。
- 不引入统一入口和规则 profile。

## 11. 风险与注意事项

- 不要让 output schema 的 Zod 泛型再次引发 typecheck OOM。
- 不要破坏现有 Markdown 输出。
- 不要把字段命名做成中文。
- 不要让 `structuredContent` 与 Markdown 内容语义不一致。
- 不要在 v0.2 顺手做 `intentRouter`，避免范围扩大。
