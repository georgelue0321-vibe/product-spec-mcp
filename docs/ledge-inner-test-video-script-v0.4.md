# 立界 Ledge 内测招募视频脚本 v0.4

## 版本判断

当前 `product-spec-mcp@0.4.2` 已经不是早期“追问 + local-first 判断”的演示状态。新版更值得拍的是：

```text
一句话产品想法
-> 立界先判断产品类型和技术边界
-> 该简单的别复杂化
-> 该上后端的别假装纯前端
-> 该本地计算的别误判成支付系统
-> 再生成规格、架构和验收标准
```

这版视频不应该把 `product_spec_connect` 作为主角。连接流程适合放在结尾“如何试用”里，用 10 秒讲清：打开连接页、下载连接文件、发回 Agent。

## 视频目标

招募 10-20 个正在用 Agent 做小产品的新手用户，尤其是有真实想法但还不会整理成工程规格的人。

核心信息：

```text
不是 Agent 不会写，
是你还没给它立界，
它就开始脑补产品和架构了。
```

## 主叙事

这次不要只拍“家庭药品管理工具”。家庭药箱仍然保留为主实录，但要加两个快速对照：

- 家庭药箱：本地小工具，不需要后端。
- 扫码点餐：订单流和后厨状态，需要轻后端。
- AA 记账：本地计算工具，不是支付系统。

这三个例子能说明新版真正的价值：不是一律让需求变复杂，也不是一律压成纯前端，而是先把边界判断对。

## 视频时长

主版建议 3 分 30 秒到 4 分 20 秒。

60 秒短版只保留：

1. 痛点开场
2. 三个需求对照
3. 家庭药箱实录
4. 招募内测

## 分镜脚本

### 01. 痛点开场：Agent 不是写不出来，是太会脑补

**时长**：0:00-0:12

**画面**：Agent 对话框快速生成文件、接口、数据库、登录页。画面里穿插“家庭药箱”“扫码点餐”“AA 记账”等用户原话。

**屏幕文案**：

```text
Agent 最可怕的不是写不出来，
而是它会很认真地脑补。

一个小工具，
可能被写成复杂系统。
```

**旁白**：

```text
现在很多人用 Agent 做产品，最大的问题不是它不会写代码，而是你还没说清边界，它就开始脑补产品、数据库、后台和接口了。
```

### 02. 点题：写代码前，先用立界

**时长**：0:12-0:25

**画面**：Logo 或简洁标题卡。不要做科技感大片，做成“开工前检查卡”。

**屏幕文案**：

```text
写代码前，
先用「立界 Ledge」。

先定边界，
再让 Agent 开工。
```

**旁白**：

```text
所以我做了一个 MCP，叫立界 Ledge。它不是替你写代码，而是在 Agent 开工前，先帮你判断这个想法到底应该怎么做第一版。
```

### 03. 简单解释 MCP

**时长**：0:25-0:38

**画面**：用户想法 -> Agent -> 立界 Ledge -> 规格/架构/验收。

**屏幕文案**：

```text
MCP = 给 AI Agent 接的外部工具

立界 Ledge =
开工前的产品边界检查器
```

**旁白**：

```text
MCP 你可以简单理解成给 AI Agent 接的外部工具。立界做的事，是把模糊想法先过一遍产品和工程边界，再交给 Agent 写代码。
```

---

### 04. 三个需求对照：同样一句话，技术路线完全不同

**时长**：0:38-1:12

**画面**：三张卡片横向切换。

**屏幕文案**：

```text
家庭药箱小网页
-> 本地工具
-> localStorage
-> 不需要后端

扫码点餐系统
-> 订单流
-> SQLite + 后厨状态 API
-> 需要轻后端

AA 记账小工具
-> 本地计算
-> participants / payments / settlements
-> 不是支付系统
```

**旁白**：

```text
比如家庭药箱小网页，第一版应该是本地工具，不需要后端。扫码点餐就不一样，顾客下单、后厨看状态、老板改价格，这需要统一后端状态源。AA 记账又是另一类，它有付款记录，但本质是本地计算，不是支付系统。
```


## 实机录制素材

### 主演示：家庭药箱

```text
我想做一个家庭药箱管理小网页，记录药品名称、有效期、数量和提醒，数据存在浏览器里
```

期望剪出：

```text
technicalProfile.shape = local_storage_tool
frontendOnly = true
needsBackend = false
needsAuth = false
suggestedStorage = localStorage

coreFeatures:
- 药品记录管理
- 新增/编辑/删除
- 到期/过期提醒
- 数量/库存管理

apiDesign:
无需 API
```

### 对照一：扫码点餐

```text
我想做一个扫码点餐系统，顾客扫码下单，后厨能看到订单状态，老板能维护菜品和价格。
```

期望剪出：

```text
technicalProfile.shape = light_backend_json_sqlite
needsBackend = true
needsAuth = true
needsAdmin = true

dataModel:
menus / dishes / orders / order_items / staff_users

apiDesign:
POST /api/orders
GET /api/kitchen/orders?status=
PATCH /api/kitchen/orders/:id/status
PATCH /api/admin/dishes/:id
```

### 对照二：AA 记账

```text
做一个 AA 记账小工具，输入每个人付了多少钱和参与人，自动算谁该转给谁，数据本地保存。
```

期望剪出：

```text
technicalProfile.shape = local_storage_tool
needsBackend = false

dataModel:
participants / payments / shareAmount / settlements

验收：
总付款金额、每人应付金额和差额计算一致
转账建议能清零差额
```

### 连接流程

录制时不要展示真实 token。

```text
product_spec_connect
-> 打开 https://productmcp.opc-mind.top/connect
-> 下载 product-spec-mcp-connect.json
-> 发回 Agent
-> Agent 写入 MCP env
```

## Remotion 制作备注

- 第一屏要痛，不要先解释 MCP。
- “立界 Ledge”只作为品牌露出，不要每个镜头都重复。
- 三个需求对照要快，像诊断报告，不要讲成长教程。
- 技术字段可以出现，但每个字段旁边要有中文解释。
- 连接流程只做轻描淡写的试用入口，不要喧宾夺主。
- 不要做成大厂产品宣传片，应该像真实工具的内测招募。

## 当前待确认点

- 主视频是否接受 4 分钟以上，还是必须压到 3 分钟内。
- 三个对照是否保留“扫码点餐 + AA 记账”，还是换成更贴近目标用户的“AI 简历网站 + 家庭药箱 + 活动报名”。
- 是否继续使用「立界 Ledge」作为主品牌名。
- 连接页是否要在第一版视频里露出域名，还是内测用户私信后单独发。
