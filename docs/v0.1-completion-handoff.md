# v0.1 版本收口交接文档

日期：2026-06-20  
目标版本：v0.1  
定位：完成 `product-spec-mcp` 的可运行 MVP，并把已修复的工程问题收口成可发布前基线。

说明：本文中的 v0.1/v0.2/v0.3 是产品路线版本，不等同于当前 `package.json` 中的 npm 包版本号。

## 1. v0.1 定位

v0.1 不追求完整商业化能力，目标是证明这个 MCP Server 的基础方向成立：

- 能作为 stdio MCP Server 运行。
- 能稳定暴露 6 个基础工具。
- 能对模糊产品需求做追问、规格编译、架构判断、UI 术语翻译、Debug 指引、验收标准生成。
- 能通过基础工程验证：类型检查、测试、构建、打包。

一句话定位：

> 面向 AI 编程用户的轻量产品规格 MCP MVP。

## 3. v0.1 版本边界

v0.1 应该保留为一个清晰、轻量的本地工具，不要继续扩大范围。

包含：

- 6 个现有 MCP tools。
- 本地 JSON 规则。
- Markdown 输出。
- 基础测试和构建脚本。
- README 中的安装、运行、客户端配置说明。

不包含：

- `structuredContent` / `outputSchema`。
- 统一入口工具。
- 外部规则 profile。
- 云端规则市场。
- 团队协作、账号、计费。
- Jira、Linear、GitHub Projects 等重集成。

## 4. v0.1 最终验收清单

交给下一个 Agent 时，建议按以下顺序验证：

```bash
npm install
npm run typecheck
npm test
npm run build
npm --cache /private/tmp/product-spec-mcp-npm-cache pack --dry-run
node dist/index.cjs
```

验收标准：

- `npm run typecheck` 不再 OOM。
- `npm test` 全部通过。
- `npm run build` 能生成 `dist/index.cjs`。
- `npm pack --dry-run` 不被本机 npm cache 权限问题阻断；如遇阻断，使用临时 cache。
- `npm pack --dry-run` 的 tarball 只包含预期发布文件。
- `node dist/index.cjs` 不抛启动期异常。
- 纯展示页/官网类项目不再推荐数据库或 API。
- 低 readiness 场景不会输出看似正式但不可执行的规格。

## 5. 发布前建议补充

v0.1 如果要对外发布，建议补齐以下轻量材料：

- README 增加“一句话适用场景”：防止 AI 编程直接从模糊需求开工。
- README 增加推荐流程：
  1. `spec_interrogate`
  2. `spec_compile`
  3. `architecture_decide`
  4. `acceptance_generate`

  - 报名系统。
  - 展示官网。
- `examples/` 中标明哪些输入适合哪个工具。
- 如果准备 npm 发布，确认 `package.json` 的 `repository`、`homepage`、`bugs`、`files`、`engines`、`bin` 都正确。
- 当前 `.gitignore` 已覆盖 `.DS_Store`，但本地已有 `.DS_Store` 文件仍应在真正建仓或发布前清理。

## 6. v0.1 风险说明

v0.1 可以作为 MVP，但不应宣传为成熟产品：

- 输出仍然主要给人读，不适合下游 agent 自动解析。
- 工具选择依赖用户或模型自己判断，没有统一入口。
- 规则还需要改源码才能调整，不适合团队定制。
- 中文口语化需求是优势，但目前还没有真实用户样例评测。

## 7. v0.1 完成定义

满足以下条件即可认为 v0.1 完成：

- 工程验证全部通过。
- README 能让新用户在 Claude Desktop 或 Cursor 中完成安装。
- 至少 2 个端到端示例可复制运行。
- `docs/project-review-report-2026-06-20.md`、本交接文档和 v0.2/v0.3 方案都存在。
- 不把 v0.2/v0.3 的能力混入 v0.1 发布口径。

## 8. 下一步版本衔接

推荐顺序：

1. v0.1：完成 MVP 收口和发布前基线。
2. v0.2：补 `structuredContent` 和 `outputSchema`，让工具结果可被 agent 稳定消费。
3. v0.3：接入 `intentRouter`，新增统一入口工具，降低用户选择工具成本。
4. v0.4：再做规则 profile 和产品化配置。
